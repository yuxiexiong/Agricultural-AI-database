## 0. 全局认知框架

### 0.1 一句话定位

农业 AI 的核心问题不是“有没有数据”，而是“能不能把分散、异构、季节性强的数据，转化成可执行、可解释、可量化 ROI 的农艺决策”。

传感器、卫星、无人机、农机、气象、作物模型、市场价格和农事记录已经产生大量数据，但农业项目最常见的断点在于：

- 数据时空尺度不一致：卫星是天级/米级，IoT 是分钟级/点位级，农机是路径级，作物表现是季节级。
- 数据权属与共享信任不足：农户、农机厂商、平台、合作社、加工企业之间难以形成可信数据流。
- 模型难以跨区域泛化：同一个病虫害、土壤水分或产量模型在不同地区、品种、栽培制度下表现会明显变化。
- 推荐难以进入真实工作流：农户/农企不需要“更复杂的模型”，而需要可解释、低成本、能直接指导当天动作的建议。

因此，本咨询项目的价值落点是：帮助甲方把农业数据资产组织成“可治理、可建模、可部署、可复用”的决策能力。

### 0.2 可用于 IP 的核心表达

- 农业 AI 不是把大模型搬到田里，而是把田里的数据变成每天能用的决策。
- 农业的难点不是缺算法，而是缺标准化、可信共享和场景闭环。
- 未来农业竞争，不只比谁有更多土地，而是比谁能把每一块地管理到有据可依。
- 2026 年之后，农业 AI 的关键词从“炫技”转向 ROI、韧性、连通性和数据主权。
- 真正可落地的农业 AI，一半是模型，一半是农艺、设备、数据治理和运营流程。

### 0.3 关键趋势判断

1. 行业从“单点工具”转向“连接系统”：单一 App、单一传感器、单一模型的价值有限，未来更重要的是数据空间、平台互操作、MLOps 和农事流程集成。
2. 从“采集数据”转向“治理数据”：欧盟农业数据空间强调数据共享、数据主权、安全和互操作；这说明数据治理已成为农业 AI 的前置能力。
3. 从“云端智能”转向“云边协同”：训练、分析、数字孪生适合云端；田间喷洒、巡检、畜舍控制、弱网环境适合边缘推理。
4. 从“自动化替代人”转向“人机协同”：开放田间环境复杂，全自动仍受限制；更现实的路线是机器完成重复、危险、耗时任务，人保留判断与责任。
5. 从“通用大模型问答”转向“农业知识 + 多源数据 + 可解释推荐”：农业 LLM 的价值不在闲聊，而在把农学知识、实时数据和本地约束组织成可靠建议。

### 1.2 技术选型四问

1. 问题类型是什么？
   - 识别/分类：优先 CV、遥感、传统机器学习。
   - 预测/预警：优先时序模型、机理模型 + 数据融合。
   - 解释/交互：优先 LLM + RAG + 知识图谱。
   - 控制/自动作业：优先边缘 AI、机器人、强化学习或规则控制。

2. 数据是否就绪？
   - 是否有足够样本？
   - 是否有标签？
   - 是否有地块、时间、作业、天气、产量等关键字段？
   - 是否能跨季节、跨区域、跨设备对齐？
   - 数据是否可追溯、可授权、可共享？

3. 落地环境有什么约束？
   - 田间是否弱网/离线？
   - 终端用户是农户、农艺师、企业管理者还是监管者？
   - 是否需要边缘低时延？
   - 是否涉及数据安全、农户隐私、商业机密或跨境数据？

4. ROI 能否被量化？
   - 降低投入品：水、肥、药、能耗。
   - 降低人工：巡检、采摘、分拣、记录、报告。
   - 提高产量或品质：增产、减损、稳定等级。
   - 满足合规：溯源、碳、农残、保险、补贴申报。
   - 提升管理效率：把经验判断转为标准化流程。

选型口诀：问题类型决定模型族，数据就绪度决定可行性，落地环境决定部署形态，ROI 决定优先级。

### 1.4 农业建模的适配原则

- 先做时空配准，再做建模：统一地块 ID、坐标系、时间粒度、作物季和作业事件。
- 先做质量分级，再做算法比较：数据按可用、需清洗、缺失、不可用分级，避免模型背锅。
- 优先小范围试点：用 1-2 个作物、1-2 个区域、1 个高频痛点验证 ROI，再扩展。
- 优先可解释模型和规则融合：农业用户更关心“为什么建议今天浇水/打药”，不是模型分数。
- 优先使用公开数据与本地数据结合：公开遥感、气象、土壤数据用于预训练或背景变量，本地数据用于校准。
- 设计跨季节验证：农业一年样本少，不能只看单季效果。

### 1.5 给甲方的三步适配方案

第一步：数据资产盘点  
形成数据源清单、字段字典、质量等级、权限归属、缺口清单。

第二步：统一数据底座  
完成地块编码、时空配准、数据治理、特征库、数据权限和可追溯机制。

第三步：场景化建模试点  
优先选择 ROI 清楚的场景，如灌溉优化、病虫害识别、变量喷洒、产量预测、农技问答、质量分级。每个试点都要定义基线、干预方式、评价指标和扩展条件。

### 2.2 技术发展趋势

#### 趋势一：农业数据空间与互操作成为基础设施

农业 AI 的瓶颈不是缺单点模型，而是缺可以跨农机、平台、作物、地块和机构流动的数据。欧盟农业数据空间、数据主权和互操作讨论说明，未来农业数据平台要具备：

- 数据授权与访问控制。
- 统一地块/作物/作业编码。
- 开放接口和标准数据模型。
- 数据血缘、审计、可追溯。
- 支持 B2B、B2G、供应链和保险等多方数据交换。

IP 讲法：谁定义农业数据的普通话，谁就更接近农业 AI 的入口。

#### 趋势二：生成式 AI 从“问答”走向“农艺解释层”

农业大模型的价值不是替代专家，而是把专家知识、项目数据、历史案例、实时监测和农事计划组织成可解释建议。可落地形态包括：

- 农技知识库问答。
- 异常诊断解释：为什么某地块缺水、虫害风险升高、长势偏弱。
- 自动生成巡田报告、月度简报、种植建议。
- 农事任务助手：提醒、计划、复盘。
- 面向管理者的自然语言 BI。

风险边界：

- 不能让通用模型直接给农药、兽药、病害处置建议，必须加入知识库、规则、地区标签、置信度和专家审核。
- 推荐内容要区分“信息解释”“建议方案”“必须人工确认的决策”。

#### 趋势三：云边协同和低成本 Edge AI

田间弱网、延迟、隐私和设备成本决定了农业 AI 不能全部依赖云端。更合理的架构是：

- 云端：训练、模型管理、跨区域分析、数字孪生、报告生成。
- 边缘：视觉识别、精准喷洒、设备控制、巡检、离线推理。
- 终端：农户 App、专家工作台、农机控制屏、语音交互。

IP 讲法：农业 AI 的省钱逻辑是训练上云、推理下沉、价值先验证、硬件后扩张。

#### 趋势四：多模态农业 AI 成为主流路线

农业天然是多模态场景：图像、遥感、气象、土壤、文本、作业记录、农机轨迹同时存在。未来更有价值的模型不是单一识别器，而是能融合：

- 卫星/无人机图像。
- 田间传感器。
- 农机作业记录。
- 天气和积温。
- 农艺文本知识。
- 市场和供应链数据。

#### 趋势五：数字孪生和预测模型进入示范项目

数字孪生适合园区、温室、规模化农场、畜舍、供应链和监管场景。它的价值不是“做一个 3D 展示”，而是：

- 模拟不同灌溉/施肥/用药策略。
- 预测产量、风险和成本。
- 支持调度和资源配置。
- 形成可复盘的管理闭环。

#### 趋势六：机器人和自动化以 ROI 为主线推进

农业机器人最强驱动力不是炫技，而是劳动力短缺、投入品成本、作业安全和标准化。当前更现实的是人机协同：

- 机器做重复劳动：巡检、喷洒、搬运、分拣、记录。
- 人做判断：是否采纳建议、异常确认、责任决策。
- 系统做闭环：记录、学习、优化、复盘。

### 2.3 行业标杆案例

| 案例 | 类型 | 可讲重点 | 咨询启示 |

### 2.4 市场需求变化

1. 需求从“尝鲜”转向“ROI”
   - 客户会问：能不能少用水、肥、药？能不能少雇人？能不能减少损失？能不能满足合规？能不能帮我卖得更好？

2. 需求从“单点识别”转向“全流程建议”
   - 单纯识别病虫害不够，用户需要：风险来源、处置建议、成本影响、最佳时间、操作记录和复盘。

3. 需求从“管理大农场”转向“兼容中小农户”
   - 小农户更需要低成本、开箱即用、少维护、可解释的方案。
   - 咨询中要区分大农企版、合作社版、小农户版，不能一个平台打所有场景。

4. 需求从“多装设备”转向“少装但有用”
   - 农业场景中，设备越多，维护越重。更好的策略是用最少必要数据验证价值，再补设备。

5. 需求从“数据上云”转向“可信共享”
   - 农户和农企会关注数据是否被平台拿走、是否会被第三方滥用、是否影响商业利益。数据授权、合同、分级和审计会变成咨询重点。

## 3. 模块三：落地实操咨询

对应合同服务：AI 技术在国际农业大数据项目中的落地实操咨询，如算法模型优化、大数据平台与 AI 技术融合应用、算力资源配置、技术落地难点解决方案等。

### 3.1 算法模型优化建议

#### 小样本与季节性

问题：农业一年只有有限生长季，同一地区同一作物的有效样本极少。  
建议：

- 使用迁移学习和公开数据预训练。
- 用作物机理模型补充纯数据模型。
- 建立跨季节评估，不以单季效果替代长期稳定性。
- 对模型输出增加置信区间和人工复核机制。

#### 区域差异与领域自适应

问题：模型在 A 地区有效，不代表在 B 地区有效。  
建议：

- 按地区、品种、土壤、栽培制度建立标签。
- 做 domain adaptation 或本地再校准。
- 保留本地基线模型，避免一刀切。

#### 多模态融合

问题：单一传感器容易误判。  
建议：

- 病害风险：图像 + 天气 + 物候 + 历史病害。
- 灌溉建议：土壤水分 + 天气预报 + 作物阶段 + 土壤类型。
- 产量预测：遥感 + 作业 + 品种 + 气象 + 历史产量。

#### 可解释与安全边界

问题：农业建议会影响产量、成本和安全。  
建议：

- 输出推荐理由、数据依据、置信度、适用条件。
- 对用药、病害处置、动物健康等高风险建议设置专家审核。
- 保留操作日志和模型版本，便于追责与复盘。

### 3.2 大数据平台与 AI 融合参考架构

```text
应用层：农技问答 / 决策建议 / 预警 / 巡田报告 / 数字孪生 / 经营看板
智能层：CV / 遥感模型 / 时序预测 / LLM + RAG / 知识图谱 / 规则引擎
工程层：MLOps / 特征库 / 模型监控 / 数据质量监控 / 权限与审计
数据层：湖仓 / 时空配准 / 主数据 / 数据血缘 / 元数据 / 数据标准
采集层：卫星 / 无人机 / IoT / 农机 / 气象 / 市场 / 农艺记录
```

关键设计原则：

- 平台不能只是“数据湖”，必须有特征库、模型闭环和数据治理。
- 模型不能只是“上线”，必须有监控、漂移检测、版本回滚和再训练机制。
- 应用不能只是“看板”，必须进入真实农事流程：提醒、执行、记录、复盘。
- 数据不能只是“汇总”，必须有权属、授权、脱敏、审计和共享规则。

### 3.3 算力资源配置策略

| 部署位置 | 适合任务 | 优势 | 风险 | 建议 |
|---|---|---|---|---|
| 云端 | 大模型、训练、跨场分析、数字孪生、报告生成 | 弹性强、集中治理、便于 MLOps | 成本、网络、数据安全 | 先用于试点和训练 |
| 边缘设备 | 喷洒、采摘、巡检、畜舍控制、弱网推理 | 低延迟、离线可用、隐私本地化 | 运维分散、硬件成本 | ROI 明确后再规模化 |
| 本地服务器 | 数据敏感、园区/企业内网、国产模型部署 | 安全可控、延迟低 | 初始投入和维护 | 适合大型农企/政府园区 |
| 移动端 | 农户问答、拍照识别、任务提醒 | 触达强、使用方便 | 机型差异、离线能力有限 | 做轻量交互，不承载重模型 |

务实路径：

1. 云端验证价值。
2. 小范围边缘试点。
3. 根据时延、网络、成本和数据安全决定是否下沉。
4. 对高频任务做轻量化，对低频复杂任务保留云端。

### 3.4 落地难点与解决方案

| 难点 | 典型表现 | 解决方案 |
|---|---|---|
| 数据孤岛 | 数据在农机厂商、平台、农户、合作社和政府系统中分散 | 建立统一数据字典、接口规范、授权机制和主数据管理 |
| 时空对齐困难 | 不同数据频率、坐标、分辨率不一致 | 建立地块 ID、作物季、作业事件和空间网格 |
| 弱网/离线 | 田间网络不稳定 | 边缘推理、离线缓存、异步同步、卫星/专网补充 |
| 小样本 | 单季数据少，标签不足 | 迁移学习、公开数据、主动学习、专家标注闭环 |
| 模型泛化差 | 换地区/品种后效果下降 | 区域化模型、本地校准、持续监控 |
| 农户不信任 | 推荐理由不清楚，怕担责 | 可解释输出、专家背书、试点对照、收益量化 |
| 设备维护重 | 传感器损坏、漂移、断电 | 设备健康监控、冗余数据源、低维护硬件 |
| ROI 难证明 | 成本先发生，收益后体现 | 设计基线和对照组，优先选每日高频痛点 |
| 合规与数据安全 | 数据权属、隐私、跨境共享不清 | 数据分级、合同授权、脱敏、审计、最小必要原则 |

### 3.5 建议试点场景

| 场景 | 试点周期 | 核心数据 | 成功指标 | 适合讲法 |
|---|---:|---|---|---|
| 智能灌溉建议 | 1 个作物季 | 土壤水分、天气、作物阶段、灌溉记录 | 节水、稳产、减少人工巡田 | 从“凭经验浇水”到“按证据浇水” |
| 病虫害图像识别 + 风险预警 | 1-2 个月起 | 图像、天气、历史病害、地块 | 识别准确率、提前预警天数、误报率 | AI 不是看图猜病，而是结合天气和农艺判断风险 |
| 变量喷洒/除草 | 1 个作业季 | 视觉、地块、作业轨迹、用药量 | 药剂节省、作业质量、杂草控制效果 | 只给需要的地方用药 |
| 产量预测 | 1 个作物季以上 | 遥感、气象、作业、历史产量 | 预测误差、提前期 | 从季末知道结果，变成季中预判风险 |
| 农技知识助手 | 4-8 周 | 知识库、FAQ、政策、农艺规程 | 命中率、人工节省、专家审核通过率 | 把专家经验变成随时可用的服务 |
| 农产品质量分级 | 1-2 个月 | 图像、等级标签、检测记录 | 分级一致性、人工节省 | 让质量判断标准化 |

## 7. 对甲方项目的建议定位

建议把项目定位为：

“面向国际农业场景的农业大数据 + AI 咨询服务，帮助甲方从数据采集、数据治理、AI 选型、场景试点到平台化落地，构建可解释、可扩展、可量化 ROI 的农业智能决策能力。”

建议第一阶段优先做三个交付：

1. 农业 AI 场景选型矩阵：按作物/地区/业务痛点筛选最优先场景。
2. 数据资产与缺口诊断：明确哪些数据可用、哪些要补、哪些不值得采。
3. 1-2 个 ROI 清楚的试点方案：如智能灌溉、病虫害预警、农技知识助手或变量喷洒。

建议传播主线：

- 对政府/园区：智慧农业、数据空间、气候韧性、产业升级。
- 对农企：降本增效、风险预警、标准化管理、供应链透明。
- 对农户/合作社：少走弯路、少浪费投入品、建议看得懂、工具用得起。
- 对国际合作方：数据治理、互操作、可持续农业、气候智慧型农业。

### 8.3 政策与监管素材

#### POL-001：全球农业增长压力与生产率主线

素材类型：政策/宏观  
核心信息：OECD-FAO Agricultural Outlook 2026-2035 预计未来十年全球农业和水产食品生产将增长约 13%，主要由生产率提升和集约化驱动；直接农业温室气体排放预计增加约 6%，慢于生产增长。  
可用角度：

- 农业 AI 的宏观价值不是“替代农民”，而是帮助农业在生产增长、成本压力和减排约束之间找到平衡。
- 生产率提升是未来十年农业政策和商业需求的共同主线。
- AI、精准农业、遥感、数据治理可以被包装成“可持续生产率工具箱”。

适合话术：未来农业不是只追求多产，而是在土地、水、肥、劳动力和碳约束下更聪明地生产。  
来源：OECD-FAO Agricultural Outlook 2026-2035，https://www.oecd.org/en/publications/oecd-fao-agricultural-outlook-2026-2035_47874669-en.html

#### POL-002：欧盟农业数字化的政策逻辑

素材类型：政策/数据治理  
核心信息：欧盟认为数字技术、IoT、传感器、AI 数据分析和决策支持系统可以提升农业可持续性、生产率和韧性；同时，数字鸿沟、技能不足、成本效益、数据共享信任和互操作不足是主要挑战。  
可用角度：

- 做国际农业大数据项目，要把技术方案和数据治理、农户培训、互操作、信任机制一起讲。
- 欧盟经验说明“农业 AI”不是单一算法采购，而是农业数字基础设施建设。

适合话术：农业 AI 的国际化表达，不应只讲模型先进，还要讲数据主权、互操作、技能培训和包容性。  
来源：European Commission - Digitalising the EU agricultural sector，https://digital-strategy.ec.europa.eu/en/policies/digitalisation-agriculture

#### POL-003：欧盟 CAP 与数字化工具

素材类型：政策/补贴方向  
核心信息：CAP 2023-2027 的关键目标之一是通过知识、创新和数字化推动农业和农村现代化；CAP 数字化工具包括宽带、精准农业投资、生态计划、农场咨询、数字技能培训和示范农场。  
可用角度：

- 海外农业 AI 项目可以对齐“现代化、环保、农民能力建设、数字基础设施”。
- 做欧盟或类欧盟政策市场时，要把项目设计成 CAP 目标可解释的能力模块。

适合话术：在欧洲语境下，农业 AI 不是孤立科技项目，而是农业现代化和农村数字基础设施的一部分。  
来源：European Commission - Digitalisation of agriculture and rural areas in the EU，https://agriculture.ec.europa.eu/overview-vision-agriculture-food/digitalisation_en

#### POL-004：FAO 气候智慧型农业三目标

素材类型：国际组织/公共价值  
核心信息：FAO 将 Climate-Smart Agriculture 定义为推动农食系统走向绿色、气候韧性实践的方法，目标包括提高农业生产力和收入、增强气候适应与韧性、尽可能减少或移除温室气体排放。  
可用角度：

- 农业 AI 项目可以对齐“产量、韧性、低碳”三目标。
- 气候智慧型农业适合作为国际化内容的公共价值框架。

适合话术：农业 AI 最有说服力的叙事不是“更酷”，而是“增产、抗风险、少排放”。  
来源：FAO Climate-Smart Agriculture，https://www.fao.org/climate-smart-agriculture/en/

#### POL-005：AIM for Climate 投资与合作网络

素材类型：国际合作/投融资  
核心信息：AIM for Climate 是美国和阿联酋联合发起的倡议，聚焦气候智慧农业和食物系统创新；官方披露 2021-2025 年动员 292 亿美元新增投资，伙伴超过 800 个，覆盖 56 个国家。  
可用角度：

- 国际资金正在关注“农业创新 + 气候 + 食物系统”。
- 农业 AI 咨询可以包装为气候智慧型农业创新服务。

适合话术：农业 AI 不只是产业升级，也正在进入全球气候投资和食物安全议程。  
来源：AIM for Climate，https://www.aimforclimate.org/

### 8.4 市场与宏观素材

#### MKT-001：十年农业需求增长与生产率压力

素材类型：市场趋势  
核心信息：OECD-FAO 2026-2035 展望指出，全球生产预计增长 13%，增长集中在亚洲、撒哈拉以南非洲和拉美；生产率提升是增长主因。  
可用角度：

- 需求增长区域和技术缺口区域高度重叠，给农业 AI 出海提供机会。
- AI 咨询可以服务“提高劳动生产率、提高资源效率、降低波动风险”三类需求。

适合输出：行业趋势页、项目背景、甲方高层汇报。

#### MKT-002：农业收入波动与风险管理

素材类型：市场风险  
核心信息：OECD-FAO 2026-2035 指出，尽管基线预测农业劳动生产率提升，但自然和经济条件波动仍可能导致农业劳动收入下降；低收入国家面临更高风险。  
可用角度：

- AI 的价值不只是“平均增产”，还包括“风险预警”和“收入韧性”。
- 灾害预警、保险、遥感评估、供应链预测是重要商业场景。

适合话术：农业经营最怕的不是某一天不够聪明，而是一季风险来临时没有提前量。

#### MKT-003：低收入地区的技术鸿沟

素材类型：发展中市场  
核心信息：OECD-FAO 2026-2035 指出，低收入地区农业劳动生产率低，机械化有限，市场接入不足，基础设施和技术缺口阻碍农民抓住增长机会。  
可用角度：

- 低成本、低维护、边缘 AI、手机端工具、合作社服务模式，比重资产智能农场更适合发展中地区。
- 国际农业 AI 不应只复制北美大型农场模式。

适合话术：在很多国家，农业 AI 的第一步不是无人农场，而是让小农户第一次拥有稳定、可解释、可负担的决策支持。

#### MKT-004：供应链损耗与市场接入

素材类型：供应链/流通  
核心信息：农业供应链中的存储、运输、市场信息、质量分级和预测不足，会造成损耗、价格波动和错配。  
可用角度：

- AI 不只用于田间，也用于仓储、冷链、分级、预测、溯源和金融。
- 如果甲方项目是“国际农业大数据”，不要只做生产侧，还应覆盖产前、产中、产后。

适合输出：价值链图、产业链延伸页、供应链应用场景页。

### 8.5 数据源与数据治理素材

#### DATA-001：农业数据资产分类

素材类型：数据底座  
分类：

- 地块数据：边界、面积、坐标、土壤、坡度、灌排条件。
- 作物数据：品种、播期、密度、生育期、长势、产量、品质。
- 环境数据：气象、土壤水分、温湿度、光照、病虫害压力。
- 作业数据：播种、施肥、喷药、灌溉、收获、机械轨迹。
- 遥感数据：卫星、无人机、植被指数、热红外、SAR。
- 经营数据：成本、价格、订单、库存、物流、销售。
- 合规数据：农残、追溯、认证、碳、保险、补贴。

使用建议：做数据资产盘点表时按“已有/缺失/低质量/无法授权”四类标注。

#### DATA-002：农业数据治理最小闭环

素材类型：数据治理 SOP  
最小闭环：

1. 地块主数据：统一地块 ID、边界和作物季。
2. 数据字典：定义字段、单位、来源、采集频率、责任人。
3. 数据质量：完整性、准确性、时效性、异常值、漂移。
4. 权限与授权：农户、企业、平台、第三方、政府的访问边界。
5. 数据血缘：从原始数据到特征、模型、报告的可追溯链路。
6. 反馈回流：农户采纳与否、实际产量/效果、专家修正进入再训练。

适合话术：没有数据治理，农业 AI 只能做演示；有数据治理，模型才有持续变好的土壤。

#### DATA-003：农业数据共享的信任问题

素材类型：数据主权/信任  
核心信息：农业数据共享常因农户担心数据被滥用、平台锁定、收益分配不清而受阻。AgriTrust 等研究把这种问题概括为农业数据价值被信任与互操作不足锁在孤岛中。  
可用角度：

- 数据共享不是技术接口问题，而是信任、合约、收益分配和合规问题。
- 项目设计中要给出数据授权、用途限制、价值回馈和审计机制。

来源：AgriTrust: a Federated Semantic Governance Framework for Trusted Agricultural Data Sharing，https://arxiv.org/abs/2511.05572

#### DATA-004：Sentinel-2 遥感数据

素材类型：公开数据源/遥感  
核心信息：Copernicus Sentinel-2 是宽幅、高分辨率、多光谱成像任务，用于土地服务，支持植被、土壤、水体监测；Sentinel 数据系统性、免费向公众和商业用户开放。Sentinel-2 含 13 个光谱波段，10m/20m/60m 空间分辨率，任务规格设计为赤道 5 天重访。  
农业用途：

- 作物长势监测。
- NDVI/NDRE 指标。
- 地块级异常检测。
- 作物类型识别。
- 灾害评估。
- 产量预测的输入变量。

来源：Copernicus Data Space - Sentinel-2，https://dataspace.copernicus.eu/data-collections/copernicus-sentinel-missions/sentinel-2

#### DATA-005：Landsat 长时间序列数据

素材类型：公开数据源/遥感  
核心信息：NASA/USGS Landsat 从 1972 年以来持续获取地表影像，是最长连续空间地表观测记录之一；NASA 表示所有 NASA 数据免费开放，Landsat 可用于作物周期变化、森林变化、城市扩张等长期变化研究。  
农业用途：

- 长周期地表变化分析。
- 历史耕地变化。
- 农业用水监测。
- 干旱与灾害历史复盘。
- 与 Sentinel-2 互补做长期基线。

来源：NASA Landsat，https://science.nasa.gov/mission/landsat/

#### DATA-006：NASA Harvest 农业监测

素材类型：机构/遥感平台  
核心信息：NASA Harvest 是 NASA 全球粮食供应与农业联盟，利用地球观测数据和公私合作提供及时、可行动的农业洞察，支持全球和美国农业韧性与决策；其重点包括政策快速评估、市场/贸易/供应链、农业风险、农业保险、早期预警、AI/ML 和数据系统。  
可用角度：

- 遥感 + AI 可服务政策、保险、贸易和早期预警，不只服务单个农场。
- 国际农业大数据项目可以参考 NASA Harvest 的“数据产品 + 决策支持 + 合作网络”模式。

来源：NASA Harvest，https://www.nasaharvest.org/

#### DATA-007：FAOSTAT

素材类型：官方统计数据源  
用途：

- 国家/地区农业生产、贸易、价格、土地、投入、粮食安全、排放等宏观数据。
- 用于报告中的国家比较、作物产量趋势、进出口格局、政策背景。

使用建议：用于宏观背景，不要直接替代企业级或地块级数据。
来源：FAOSTAT，https://www.fao.org/faostat/en/

#### DATA-008：SoilGrids/土壤数据

素材类型：公开土壤数据源  
用途：

- 作物适宜性分析。
- 产量预测背景变量。
- 灌溉施肥模型输入。
- 低成本地区初步建模。

使用建议：公开土壤栅格适合做背景层和初筛，不应替代本地土壤采样。
来源：SoilGrids，https://soilgrids.org/

### 8.6 标准与互操作素材

#### STD-001：ISO 11783 / ISOBUS

素材类型：农机通信标准  
核心信息：ISO 11783-1:2017 定义农业/林业拖拉机与机具之间的串行数据网络，用于传感器、执行器、控制元件、信息存储和显示单元之间的数据传输方法与格式标准化。ISO 页面显示该版本 2023 年确认有效。  
可用角度：

- 智能农机、变量作业、喷洒、播种、收获数据接入，应考虑 ISOBUS/ISO 11783 兼容。
- 农业大数据平台如果不考虑农机标准，后续会遇到设备接入和作业数据碎片化问题。

来源：ISO 11783-1:2017，https://www.iso.org/standard/57556.html

#### STD-002：ISOBUS 数据字典

素材类型：标准数据字典  
核心信息：ISOBUS 在线数据库提供 ISO 11783-11 数据字典，包括 Process Data DDI 等实体和导出文件；页面显示数据字典版本为 2026050501。  
可用角度：

- 做农机数据接入时，可把标准数据字典作为字段映射和接口设计参考。
- 适合在报告中证明“农业数据标准化不是抽象口号，而是有行业数据字典支撑”。

来源：ISOBUS Data Dictionary，https://www.isobus.net/isobus/

#### STD-003：AgGateway ADAPT

素材类型：精准农业数据互操作  
核心信息：ADAPT 全称 Agricultural Data Application Programming Toolkit，目标是解决精准农业数据在不同软硬件系统之间难以互通的问题；其网站指出多个系统采集/分析田间作业数据，但行业一直难以让不同系统用共享数据格式“对话”。ADAPT Standard 第一版于 2024 年 6 月发布。  
可用角度：

- 用于说明甲方平台要避免厂商锁定，需提前设计数据模型和转换插件。
- 适合做“农业数据普通话”的标准案例。

来源：ADAPT Framework，https://adaptframework.org/

#### TECH-001：农业 LLM 的正确架构

素材类型：技术架构  
推荐架构：

```text
→ 意图识别
→ 地区/作物/生育期/用户身份补全
→ RAG 检索：农技知识库 + 政策 + 本地 SOP + 历史案例
→ 工具调用：天气、地块、传感器、遥感、作业记录
→ 规则校验：农药/兽药/安全/合规/禁用项
→ 生成建议：解释 + 依据 + 置信度 + 风险提示
→ 专家审核/用户反馈
→ 日志与知识库更新
```

核心观点：农业 LLM 不是一个聊天框，而是“农业知识、实时数据、规则约束和专家流程”的自然语言入口。

#### TECH-002：农业知识图谱的实体关系

素材类型：知识工程  
实体：

- 作物、品种、地块、土壤、气象、物候、病害、虫害、杂草、农药、肥料、农机、作业、产量、品质、市场、政策。

关系：

- 作物-易感病害。
- 病害-触发条件。
- 虫害-发生期。
- 农药-适用对象/禁用条件/安全间隔期。
- 品种-适宜区域。
- 土壤-施肥建议。
- 气象-风险事件。
- 作业-产量/成本影响。

可用角度：知识图谱适合解决“生成式 AI 胡说”和“推荐理由不可解释”的问题。

#### TECH-003：遥感 AI 技术路径

素材类型：技术路线  
路径：

1. 数据获取：Sentinel-2、Landsat、无人机、SAR。
2. 预处理：云掩膜、辐射校正、几何校正、地块裁剪。
3. 特征构建：NDVI、NDRE、EVI、LAI、纹理、时序曲线。
4. 建模：随机森林、XGBoost、CNN、Transformer、时序模型、作物机理模型融合。
5. 输出：长势图、异常图、产量预测、灾害评估、风险分区。
6. 验证：地面样本、历史产量、专家巡田、抽样测产。

关键风险：云覆盖、地块边界不准、地面真值不足、跨年泛化差。

#### TECH-004：机理模型 + AI 融合

素材类型：建模策略  
核心观点：农业不同于纯互联网推荐，作物生长受物理、生理和环境约束；机理模型能提供可解释边界，AI 能做本地校准和复杂模式捕捉。  
适用场景：

- 产量预测。
- 灌溉施肥。
- 物候预测。
- 极端天气影响评估。
- 温室环境控制。

适合话术：农业 AI 最稳的路线，不是让模型凭空“猜”，而是让模型站在农学规律上学习本地差异。

#### TECH-005：低成本边缘 AI 路线

素材类型：技术路线/普惠农业  
方案：

- 手机端拍照识别 + 云端复核。
- ESP32/STM32/低功耗传感器做环境采集。
- 树莓派/Jetson/国产边缘盒子做本地视觉推理。
- 离线缓存 + 异步同步。
- 模型量化、剪枝、蒸馏。
- 通过合作社或服务站集中维护。

适合场景：弱网、低预算、小农户、偏远地区、试点前期。

### 8.8 标杆案例素材

#### CASE-001：John Deere See & Spray

素材类型：企业案例/精准喷洒  
核心信息：John Deere 官方说明 See & Spray Gen 2 使用相机视觉和机器学习区分作物与杂草，只喷杂草，并提供杂草压力图、株数等田间洞察。  
可讲结构：

1. 痛点：传统全面喷洒浪费药剂，也增加环境压力。
2. 数据：摄像头实时识别作物与杂草。
3. 模型：机器学习区分目标。
4. 执行：喷头只对杂草位置喷洒。
5. 价值：降低投入、提高作业精度、生成田间数据。

适合话术：这不是“给农机加 AI”，而是让每一次喷洒都变成一次数据驱动的决策。  
来源：John Deere See & Spray Gen 2，https://www.deere.com/en/sprayers/see-spray-gen-2/

#### CASE-002：CropX 数字农艺平台

素材类型：企业案例/传感器 + 农艺模型  
核心信息：CropX 官方称其持续采集并分析田间传感器、卫星和农机等数据，应用农艺模型、数据科学和 AI，预测水分、作物胁迫、病害风险、硝态氮淋失、盐分、长势、根系和冠层发育，并转化为灌溉、病害防护等建议。  
可讲结构：

1. 数据来自土壤、天气、卫星、农机。
2. 模型不仅是 AI，也包括农艺模型。
3. 输出不是图表，而是“什么时候浇、浇多少、怎么防病”的行动建议。

适合话术：农业 AI 的商业闭环，是从传感器到模型，再到每天能执行的农艺建议。  
来源：CropX，https://cropx.com/

#### CASE-003：欧盟农业数据空间

素材类型：政策基础设施案例  
核心信息：欧盟农业数据空间目标是为农业数据建立安全可信共享空间，让农业部门共享和访问数据，提升田间经济和环境绩效。  
可讲结构：

1. 问题：数据在农户、机械、平台、公共部门之间割裂。
2. 机制：数据空间强调主权、信任、安全、互操作。
3. 价值：支持精准农业、合规、补贴、环保和公共治理。

适合话术：农业 AI 的上层是模型，底层是数据空间。没有数据空间，模型很难跨主体协同。  
来源：Common European data spaces for agriculture and mobility，https://digital-strategy.ec.europa.eu/en/library/common-european-data-spaces-agriculture-and-mobility

#### CASE-004：NASA Harvest

素材类型：国际公共数据案例  
核心信息：NASA Harvest 利用地球观测数据、AI/ML、数据系统和伙伴网络，服务农业政策快速评估、市场贸易、农业保险、风险评估和早期预警。  
可讲结构：

1. 数据：卫星地球观测。
2. 方法：AI/ML + 数据系统。
3. 服务对象：政策制定者、市场、保险、农户和供应链。
4. 价值：提升粮食安全和农业韧性。

适合话术：遥感农业的最高价值，不只是看田长得好不好，而是在粮食安全、贸易、保险和灾害响应中提供提前量。  
来源：NASA Harvest，https://www.nasaharvest.org/

#### CASE-005：Landsat 长期农业监测

素材类型：公开数据案例  
核心信息：Landsat 是 50 多年的地表观测基础设施，NASA 说明其数据可用于作物周期变化、森林转变等长期变化研究，并且数据免费开放。  
可讲结构：

1. 长期数据形成基线。
2. 基线支持异常识别。
3. 异常识别支持灾害、风险、保险和政策评估。

适合话术：短期数据告诉你今天发生了什么，长期数据告诉你今天是不是异常。  
来源：NASA Landsat，https://science.nasa.gov/mission/landsat/

### 8.9 应用场景素材

#### SCN-001：智能灌溉

适合客户：规模化种植、设施农业、缺水地区、果园。  
数据需求：土壤水分、天气预报、作物阶段、土壤类型、灌溉记录、地块边界。  
模型路线：规则 + 作物需水模型 + 时序预测 + 本地校准。  
输出：灌溉时间、灌溉量、风险提示、节水复盘。  
验收指标：节水比例、产量稳定性、人工巡检减少、误报率。

#### SCN-002：病虫害识别与预警

适合客户：果蔬、经济作物、植保服务、合作社。  
数据需求：图片、天气、历史病害、作物阶段、地区病虫害历。  
模型路线：CV + 风险模型 + 专家规则 + LLM 解释。  
输出：疑似病虫害、置信度、处置建议、是否需专家确认。  
验收指标：识别准确率、提前预警天数、误报/漏报、专家审核通过率。

#### SCN-003：变量喷洒/精准除草

适合客户：大型农场、农机服务商、植保企业。  
数据需求：实时影像、地块边界、作业轨迹、药剂记录、杂草分布。  
模型路线：边缘 CV + 喷头控制 + 作业记录回传。  
输出：精准喷洒、药剂用量、杂草压力地图、作业报告。  
验收指标：药剂节省、作业覆盖率、杂草控制效果、设备稳定性。

#### SCN-004：产量预测

适合客户：农企、保险、粮食贸易、政府监管。  
数据需求：遥感、天气、土壤、品种、作业、历史产量、采样测产。  
模型路线：遥感时序 + 机器学习 + 作物机理模型 + 地面校准。  
输出：地块/区域产量预测、置信区间、风险因素解释。  
验收指标：预测误差、提前期、跨年稳定性。

#### SCN-005：农产品质量分级

适合客户：加工厂、合作社、品牌农产品、电商供应链。  
数据需求：图像、等级标签、检测记录、缺陷样本、产地批次。  
模型路线：CV 检测/分类 + 规则阈值 + 质检记录。  
输出：等级、缺陷类型、分拣建议、批次报告。

#### SCN-006：农技知识助手

模型路线：LLM + RAG + 知识图谱 + 工具调用 + 安全规则。

模型路线：遥感异常检测 + 灾害暴露评估 + 风险评分。  
输出：损失评估、理赔辅助、风险地图、早期预警。

### 8.11 风险与反方观点素材

#### RISK-001：数据不统一，模型效果不可复用

表现：换设备、换农场、换地区后模型表现下降。  
回应：先做地块主数据和数据字典，再做模型；试点要包含跨地块和跨时间验证。

#### RISK-002：传感器越多，维护越重

表现：设备断电、漂移、损坏、无人维护，数据质量下降。  
回应：最少必要设备原则；先用公开数据 + 关键传感器验证价值，再扩设备。

#### RISK-003：只做看板，不进工作流

表现：管理层看着漂亮，农户和农艺师不用。  
回应：每个模型输出必须绑定一个动作，如浇水、打药、巡田、复核、记录。

#### RISK-004：大模型幻觉和高风险建议

表现：模型给出错误农药、错误剂量、错误诊断。  
回应：RAG + 规则库 + 禁用项 + 置信度 + 专家审核；高风险建议必须人工确认。

#### RISK-005：ROI 证明周期太长

表现：农业要等一季甚至一年才看收益，客户耐心不足。  
回应：拆成短周期指标和长周期指标；先看数据完整率、建议采纳、人工节省，再看产量。

#### RISK-006：小农户用不起、不会用

表现：复杂平台、昂贵硬件、培训成本高。  
回应：手机端、合作社代运营、轻量化硬件、语音/图片交互、线下服务站。

#### RISK-007：数据权属不清

表现：农户不愿上传数据，平台与农机厂商互不开放。  
回应：合同明确数据权属、用途、期限、撤回权、收益分享和审计机制。

### 8.12 可直接改写的话术素材

#### COPY-001：一句话开场

农业 AI 的价值，不是让机器替农民思考，而是把散落在土地、天气、农机和市场里的数据，变成每天能执行的决策。

#### COPY-002：问题开场

过去农业数字化最大的问题，不是数据太少，而是数据太散：传感器有数据，农机有数据，卫星有数据，农户有经验，但它们很少在同一个决策里相遇。

#### COPY-003：政策型表达

从智慧农业、气候韧性到农业数据空间，国际政策共同指向一件事：农业要用更少资源、更强韧性，生产更多、更稳定、更可追溯的食物。

#### COPY-004：大模型表达

农业大模型不能只是一个会聊天的窗口。真正有用的农业大模型，必须连接知识库、地块数据、天气、作业记录和专家规则，最后输出可解释、可复核、可执行的建议。

#### COPY-005：数据治理表达

如果没有统一地块 ID、作物季和作业记录，农业 AI 很容易变成“看起来聪明、落地混乱”的演示系统。

#### COPY-006：遥感表达

卫星不是替代巡田，而是让巡田更有方向：先从天上发现异常，再让人和机器去最需要的地方。

#### COPY-007：普惠小农表达

对小农户来说，最好的 AI 不一定是最强模型，而是便宜、稳定、看得懂、用得上的建议。

#### COPY-008：ROI 表达

农业 AI 的 ROI 不应只问“模型准不准”，还要问：少用了多少水肥药，少跑了多少趟田，提前发现了多少风险，减少了多少损失。

#### COPY-009：国际化表达

国际农业 AI 的竞争，不只是模型能力竞争，更是数据标准、场景理解、农艺知识和本地化服务能力的竞争。

#### COPY-010：咨询交付表达

我们的咨询不是卖一个算法，而是帮助甲方把农业数据从采集、治理、建模到应用闭环串起来，形成可扩展、可验证、可持续迭代的决策能力。

### 8.13 图表与 PPT 页面素材

#### VIS-001：农业 AI 价值链图

图名：从数据到决策的最后一公里  
结构：

```text
数据源：卫星 / 无人机 / IoT / 农机 / 气象 / 农艺 / 市场
→ 数据治理：地块 ID / 时空配准 / 数据质量 / 权限 / 标准
→ AI 模型：CV / 遥感 / 时序 / LLM / 知识图谱 / 机理融合
→ 决策应用：灌溉 / 施肥 / 喷洒 / 预警 / 产量 / 溯源
→ ROI：节水 / 节肥 / 节药 / 减损 / 增产 / 合规
```

#### VIS-002：技术选型矩阵

横轴：数据就绪度低 → 高  
纵轴：ROI 清晰度低 → 高  
象限：

- 高数据 + 高 ROI：优先试点。
- 低数据 + 高 ROI：先补数据。
- 高数据 + 低 ROI：作为平台能力储备。
- 低数据 + 低 ROI：暂缓。

#### VIS-003：农业 AI 五层架构图

五层：

1. 采集层。
2. 数据治理层。
3. 模型智能层。
4. 应用决策层。
5. 运营反馈层。

关键标注：MLOps、数据血缘、专家审核、用户反馈贯穿全层。

#### VIS-004：ROI 仪表盘

模块：

- 投入品节省：水、肥、药、能耗。
- 生产表现：产量、品质、损失。
- 运营效率：人工、巡检、报告。
- 风险控制：预警提前期、异常发现、保险评估。
- 合规：追溯完整率、报告自动化率。

#### VIS-005：农业数据治理闭环图

```text
数据采集
→ 数据质量检查
→ 时空配准
→ 特征库
→ 模型训练/推理
→ 决策建议
→ 用户反馈
→ 数据更新与再训练
```

### 8.14 落地运营素材

#### OPS-001：试点到规模化路线

阶段：

1. 需求定义：明确作物、地区、业务痛点、ROI。
2. 数据盘点：数据源、质量、权限、缺口。
3. 小样本试点：1 个区域、1 个作物、1 个场景。
4. 基线对照：和传统方法对比。
5. 复盘优化：数据、模型、流程、培训。
6. 扩区扩作物：验证泛化。
7. 平台化运营：MLOps、客服、农艺师、持续更新。

#### OPS-002：月度情报服务 SOP

每月固定流程：

1. 政策扫描：官方/机构来源优先。
2. 技术扫描：论文、企业发布、开源模型、标准进展。
3. 案例扫描：国际/国内各 2-3 个。
4. 甲方相关性筛选：和甲方业务无关的只入库，不进简报。
5. 形成简报：每条包含“发生了什么、为什么重要、甲方能做什么、来源”。
6. 素材入库：按编号追加到第 8 章。

#### OPS-003：农业 AI 项目角色配置

角色：

- 项目负责人：目标、预算、验收。
- 农艺专家：知识、规则、建议审核。
- 数据工程师：数据接入、治理、质量。
- 算法工程师：模型训练、评估、部署。
- 平台工程师：系统、接口、安全、MLOps。
- 运营培训：农户/农艺师培训、反馈收集。
- 合规负责人：数据权属、隐私、用药安全、跨境。

关键提醒：农业 AI 项目不能只配算法工程师，否则最容易卡在数据和现场流程。
